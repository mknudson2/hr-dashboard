## Executive Summary

The HR Dashboard implements defense-in-depth security architecture following OWASP Top 10 guidelines, NIST 800-63B authentication standards, and industry best practices for HRIS applications. This document details the technical implementation of security controls protecting employee personally identifiable information (PII), compensation data, medical leave information, and other sensitive HR data.

### Key Security Features

| Control Category | Implementation | Standard Alignment |
|-----------------|----------------|-------------------|
| Authentication | JWT + TOTP 2FA + Backup Codes | NIST 800-63B AAL2 |
| Authorization | Fine-grained RBAC (65+ permissions) | NIST AC-2, AC-3 |
| Encryption at Rest | AES-128-CBC (Fernet) field-level | NIST 800-111 |
| Encryption in Transit | TLS 1.2+ (HTTPS enforced) | NIST 800-52 |
| Password Policy | 12+ chars, complexity, history | NIST 800-63B |
| Session Management | Server-side with idle timeout | OWASP Session Management |
| CSRF Protection | Double-submit cookie pattern | OWASP CSRF Prevention |
| Rate Limiting | Per-endpoint throttling | OWASP API Security |
| Audit Logging | Comprehensive with PII redaction | SOC 2, HIPAA |

## Authentication Architecture

### JWT Token Implementation

**Location:** `backend/app/api/auth.py`

```python
# Token Configuration
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_HOURS = 24  # Configurable via environment
```

**Token Structure:**
- **Header:** Algorithm (HS256), Type (JWT)
- **Payload:** Subject (username), Role, Expiration timestamp
- **Signature:** HMAC-SHA256 with secret key (minimum 64 characters required)

**Token Storage:**
- Primary: httpOnly cookies (prevents JavaScript access)
- Portal-specific cookies: `portal_access_token` (Employee Portal), `hr_access_token` (HR Portal)
- Bearer token support for API integrations

**Token Lifecycle:**
1. Generated on successful authentication
2. Validated on each request via dependency injection
3. Refreshed via `/auth/refresh` endpoint
4. Blacklisted on logout (prevents reuse)
5. Automatic cleanup of expired blacklist entries

### Multi-Factor Authentication (2FA)

**Implementation:** Time-based One-Time Password (TOTP) using PyOTP library

**Configuration:**
- Algorithm: SHA-1 (RFC 6238 compliant)
- Digit count: 6
- Time step: 30 seconds
- Issuer: Configurable ("HR Portal" default)

**Endpoints:**
| Endpoint | Method | Rate Limit | Description |
|----------|--------|------------|-------------|
| `/auth/2fa/setup` | POST | 3/min | Initiates 2FA setup, returns QR code |
| `/auth/2fa/verify` | POST | 5/min | Verifies TOTP code |
| `/auth/2fa/disable` | POST | 3/min | Disables 2FA (requires password) |
| `/auth/2fa/status` | GET | - | Returns 2FA enrollment status |
| `/auth/2fa/regenerate-backup-codes` | POST | 3/min | Generates new backup codes |
| `/auth/admin/reset-2fa/{user_id}` | POST | 10/min | Admin 2FA reset |

**Backup Codes:**
- 10 single-use codes generated during setup
- Stored as bcrypt hashes (cost factor 12)
- Invalidated after use
- Can be regenerated by user

### Account Lockout Protection

**Configuration:**
- Maximum failed attempts: 5
- Lockout duration: 15 minutes
- Automatic reset on successful authentication

**Database Fields:**
```python
failed_login_attempts = Column(Integer, default=0)
locked_until = Column(DateTime, nullable=True)
```

**Implementation Flow:**
1. Failed login increments `failed_login_attempts`
2. On 5th failure, `locked_until` set to current time + 15 minutes
3. Login attempts during lockout return 423 (Locked)
4. Successful login resets counter and clears lockout

## Authorization & Access Control

### Role-Based Access Control (RBAC)

**Location:** `backend/app/services/rbac_service.py`

**Architecture:**
```
Users ──┬──> UserRole ──> Roles ──> RolePermission ──> Permissions
        └──> Legacy role column (fallback)
```

**Permission Granularity (65+ permissions):**

| Category | Permissions |
|----------|-------------|
| User Management | `users:read`, `users:create`, `users:update`, `users:delete`, `users:reset_password` |
| Employee Data | `employees:read:all`, `employees:read:team`, `employees:read:self`, `employees:write:all`, `employees:write:team` |
| Compensation | `compensation:read:all`, `compensation:read:team`, `compensation:read:self`, `compensation:write` |
| FMLA | `fmla:read`, `fmla:write`, `fmla_portal:employee`, `fmla_portal:supervisor` |
| Garnishments | `garnishments:read`, `garnishments:write`, `garnishment_portal:employee` |
| Performance | `performance:read:all`, `performance:read:team`, `performance:read:self`, `performance:write:all` |
| PTO | `pto:read:all`, `pto:read:team`, `pto:read:self`, `pto:write:all`, `pto:write:team` |
| Audit | `audit:read` |

**Permission Checking Methods:**
```python
# Single permission required
@router.get("/endpoint")
def endpoint(user = Depends(require_permission(Permissions.EMPLOYEES_READ_ALL))):
    pass

# Any of multiple permissions
@router.get("/endpoint")
def endpoint(user = Depends(require_any_permission(
    Permissions.EMPLOYEES_READ_ALL,
    Permissions.EMPLOYEES_READ_TEAM
))):
    pass

# All permissions required
@router.get("/endpoint")
def endpoint(user = Depends(require_all_permissions(
    Permissions.FMLA_READ,
    Permissions.FMLA_WRITE
))):
    pass
```

### Portal Isolation

| Portal | Access Control | Data Scope |
|--------|---------------|------------|
| HR Portal | Full administrative access | All employee data |
| Employee Portal | Self-service only | Own data + limited team view |
| FMLA Portal | Medical leave management | FMLA cases (own/team) |
| Garnishment Portal | Garnishment viewing | Own garnishments only |

**Implementation:**
- `X-Portal-Source` header identifies requesting portal
- `allowed_portals` JSON array in User model restricts portal access
- Login endpoint validates portal access before issuing token

## Data Protection & Encryption

### Field-Level Encryption

**Location:** `backend/app/services/encryption_service.py`

**Algorithm:** Fernet (AES-128-CBC + HMAC-SHA256)
- Encryption: AES-128 in CBC mode
- Authentication: HMAC-SHA256
- Key derivation: PBKDF2-SHA256 (480,000 iterations per OWASP recommendations)

**Encrypted Fields:**
| Model | Fields |
|-------|--------|
| Employee | `wage`, `annual_wage`, `hourly_wage`, `benefits_cost`, `benefits_cost_annual`, `employer_taxes_annual`, `total_compensation` |
| WageHistory | `wage` |
| ACA1095CRecord | `employee_ssn` |

**Key Management:**
```bash
# Environment variables
FIELD_ENCRYPTION_KEY=<44-character Fernet key>
ENCRYPTION_SALT=<random salt for key derivation>
```

**Production Requirements:**
- Application fails to start without encryption key in production mode
- Development mode generates warning and temporary key
- Key rotation supported via re-encryption migration

**Encryption Service Methods:**
```python
# Encrypt string data
ciphertext = encryption_service.encrypt("sensitive_data")

# Encrypt numeric data
encrypted_wage = encryption_service.encrypt_float(75000.00)

# Decrypt data
plaintext = encryption_service.decrypt(ciphertext)

# Check if value is encrypted
is_encrypted = encryption_service.is_encrypted(value)
```

### Data in Transit

**HTTPS Enforcement:**
- HSTS header in production: `max-age=31536000; includeSubDomains; preload`
- TLS 1.2+ required (configured at reverse proxy/load balancer)
- Secure cookie flag enabled in production

## Input Validation & Injection Prevention

### SQL Injection Prevention

**Implementation:** SQLAlchemy ORM with parameterized queries

**Example (Safe Pattern):**
```python
# All queries use ORM - parameters are automatically escaped
users = db.query(models.User).filter(
    or_(
        models.User.username.like(f"%{search}%"),
        models.User.email.like(f"%{search}%")
    )
).all()
```

**Verification:**
- No raw SQL execution in application code
- All database operations through SQLAlchemy ORM
- Migrations use Alembic with parameterized DDL

### Request Validation

**Backend (Pydantic):**
```python
class UserCreate(BaseModel):
    username: str = Field(..., min_length=3, max_length=50)
    email: EmailStr
    password: str = Field(..., min_length=12)
    role: str = Field(..., pattern="^(admin|manager|employee)$")

    @field_validator('username')
    def validate_username(cls, v):
        if not re.match(r'^[a-zA-Z0-9_]+$', v):
            raise ValueError('Username must be alphanumeric')
        return v
```

**Frontend (TypeScript):**
```typescript
// Location: src/utils/validation.ts
export function sanitizeString(input: string): string {
  return input
    .replace(/[<>]/g, '')  // Remove angle brackets
    .trim()
    .replace(/\s+/g, ' '); // Normalize whitespace
}
```

## Session Management

### Server-Side Sessions

**Location:** `backend/app/db/models.py` (Session model)

**Session Table Schema:**
```python
class Session(Base):
    id = Column(Integer, primary_key=True)
    user_id = Column(Integer, ForeignKey("users.id"))
    token = Column(String, unique=True, index=True)
    expires_at = Column(DateTime)
    created_at = Column(DateTime, default=datetime.utcnow)
    last_activity = Column(DateTime, default=datetime.utcnow)
    ip_address = Column(String)
    user_agent = Column(String)
```

### Session Configuration

| Parameter | Default | Environment Variable |
|-----------|---------|---------------------|
| Idle Timeout | 30 minutes | `SESSION_IDLE_TIMEOUT_MINUTES` |
| Max Lifetime | 8 hours | `SESSION_MAX_LIFETIME_HOURS` |
| Cookie Secure | True (prod) | Automatic based on ENVIRONMENT |
| Cookie SameSite | Strict | Lax in development |

### Session Lifecycle

1. **Creation:** On successful login, session created with user context
2. **Validation:** Each request validates token and checks expiration
3. **Sliding Window:** `last_activity` updated on each valid request
4. **Idle Timeout:** Session invalidated if `last_activity` exceeds threshold
5. **Termination:** Explicit logout or max lifetime exceeded
6. **Cleanup:** Background process removes expired sessions

## Cross-Site Request Forgery (CSRF) Protection

### Double-Submit Cookie Pattern

**Location:** `backend/app/services/csrf_service.py`

**Implementation:**
1. CSRF token set as cookie (readable by JavaScript)
2. Frontend sends token in `X-CSRF-Token` header
3. Backend validates cookie value matches header value

**Token Generation:**
```python
def generate_csrf_token() -> str:
    timestamp = str(int(time.time()))
    random_bytes = secrets.token_hex(32)
    data = f"{timestamp}.{random_bytes}"
    signature = hmac.new(
        SECRET_KEY.encode(),
        data.encode(),
        hashlib.sha256
    ).hexdigest()
    return f"{data}.{signature}"
```

**Token Validation:**
- Signature verification using HMAC-SHA256
- Timestamp validation (max age: 8 hours)
- Cookie-header value comparison

**Protected Methods:** POST, PUT, PATCH, DELETE

**Exempt Paths:**
- `/auth/login` (no session yet)
- `/auth/refresh` (uses existing token)
- `/auth/csrf-token` (token retrieval)
- `/portal/*` (auth-protected, different flow)

## Audit Logging & Monitoring

### Audit Service

**Location:** `backend/app/services/audit_service.py`

### Event Categories

| Category | Events |
|----------|--------|
| Authentication | LOGIN_SUCCESS, LOGIN_FAILED, LOGOUT, TOKEN_REFRESH, SESSION_EXPIRED |
| Password | PASSWORD_CHANGE, PASSWORD_RESET, PASSWORD_RESET_REQUEST |
| 2FA | 2FA_ENABLED, 2FA_DISABLED, 2FA_VERIFY_FAILED, 2FA_ADMIN_RESET |
| User Management | USER_CREATED, USER_UPDATED, USER_DELETED, ROLE_CHANGED |
| Data Access | DATA_VIEW_SENSITIVE, DATA_EXPORT, BULK_DATA_ACCESS |
| Data Modification | DATA_CREATE, DATA_UPDATE, DATA_DELETE |
| Security | RATE_LIMIT_EXCEEDED, SUSPICIOUS_ACTIVITY, ACCESS_DENIED |

### Audit Log Schema

```python
class SecurityAuditLog(Base):
    id = Column(Integer, primary_key=True)
    event_type = Column(String, index=True)
    event_category = Column(String)
    severity = Column(String)  # INFO, WARNING, CRITICAL

    # Actor information
    user_id = Column(Integer, ForeignKey("users.id"))
    username = Column(String)
    ip_address = Column(String)
    user_agent = Column(String)

    # Resource information
    resource_type = Column(String)
    resource_id = Column(String)

    # Change tracking
    old_value = Column(Text)  # JSON, sensitive fields redacted
    new_value = Column(Text)  # JSON, sensitive fields redacted

    # Request context
    request_path = Column(String)
    request_method = Column(String)

    # Status
    success = Column(Boolean)
    error_message = Column(String)

    created_at = Column(DateTime, default=datetime.utcnow, index=True)
```

### Sensitive Field Redaction

**Automatically Redacted Fields:**
- SSN, Social Security
- Wages, salary, compensation
- Bank account numbers
- Passwords, tokens, secrets
- Medical information, FMLA reasons
- Personal addresses, phone numbers

**Redaction Format:** Shows last 4 characters (e.g., `***-**-1234`)

## Password Policy & Management

### Password Requirements (NIST 800-63B Aligned)

| Requirement | Value |
|-------------|-------|
| Minimum length | 12 characters |
| Maximum length | 128 characters |
| Uppercase required | At least 1 |
| Lowercase required | At least 1 |
| Digit required | At least 1 |
| Special character required | At least 1 |
| Common password check | 100+ blocked passwords |
| Sequential characters | Blocked (e.g., "1234", "abcd") |
| Repeated characters | Blocked (e.g., "aaaa") |
| Username in password | Blocked |
| Password history | Last 12 passwords blocked |

### Password Storage

- **Algorithm:** bcrypt
- **Cost factor:** 12
- **Implementation:** Passlib library with automatic algorithm upgrade support
